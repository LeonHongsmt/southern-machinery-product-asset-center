# 前端资料展示研究

本文档研究产品资料在前端页面中的展示方式，重点覆盖产品列表、产品详情、资料链接、搜索筛选、图片兜底、静态 JSON 读取方式，以及后续适合采用的页面结构。本文只做展示与信息架构研究，不包含任何业务代码实现。

## 1. 产品列表页怎么展示

产品列表页应承担“快速浏览、比较、进入详情”的作用。建议采用响应式网格或列表布局：桌面端可使用多列卡片，移动端改为单列纵向排列。每个产品条目不宜堆叠过多信息，应优先展示能帮助用户判断是否点击的核心资料。

建议每个产品卡片包含：

- 产品主图或占位图。
- 产品名称。
- 简短描述或摘要。
- 产品分类、品牌、型号等关键标签。
- 可用资料提示，例如“PDF”“图片”“说明文档”。
- 进入详情页的明确入口。

列表页还应保留资料完整度提示，例如“资料齐全”“缺少 PDF”“图片待补充”等。这类状态可以帮助运营或内部维护人员快速发现数据缺口，也能让用户提前知道详情页中可能缺少哪些内容。

排序方式建议先保持简单，可支持默认排序、按名称排序、按更新时间排序、按分类聚合排序。若产品数量较多，列表页顶部应固定搜索与筛选区域，避免用户滚动后失去操作入口。

## 2. 产品详情页怎么展示

产品详情页应围绕“完整解释一个产品”组织信息。推荐把页面分成清晰的信息区块，而不是把所有字段平铺在同一个长段落中。

建议详情页包含以下模块：

- 顶部概览：产品名称、主图、核心摘要、分类标签、资料状态。
- 基础信息：品牌、型号、规格、适用场景、来源、更新时间等结构化字段。
- 图片资料：展示主图和补充图片，可支持缩略图浏览。
- 文档资料：集中展示 PDF、说明文档、外部链接和下载入口。
- 产品说明：展示长文本介绍、使用说明、注意事项等。
- 相关产品：同分类、同品牌或同用途的推荐入口。

详情页的信息层级应遵循“先判断、再深入”的原则。用户进入页面后，第一屏应能确认产品是否是目标对象；再往下才查看参数、文档和补充资料。

对资料来源要有清楚标注。若某些数据来自静态 JSON、人工整理或外部文件链接，前端展示时应保留来源字段、更新时间字段和可追溯链接，便于后续校验。

## 3. PDF、图片、说明文档链接如何展示

资料链接不建议散落在详情页各处，最好集中在“资料与下载”模块中展示。模块内可按资料类型分组，降低用户寻找成本。

建议分组方式：

- PDF：产品手册、规格书、认证文件、宣传册。
- 图片：主图、细节图、包装图、应用场景图。
- 说明文档：安装说明、使用说明、维护说明、常见问题。
- 外部链接：官网页面、供应商页面、云盘文件、在线文档。

每个资料条目建议展示：

- 资料名称。
- 资料类型。
- 文件格式或链接类型。
- 文件大小，如果静态数据中有记录。
- 更新时间或采集时间。
- 打开、预览或下载入口。

PDF 链接适合提供“在线预览”和“下载”两个入口。如果后续前端不准备接入 PDF 预览组件，也可以先采用浏览器新标签打开方式。图片链接可在详情页内直接预览，点击后进入大图查看。说明文档链接如果是网页，可直接打开；如果是 Markdown、TXT、DOCX 等文件，应明确显示格式，避免用户误以为一定能在线阅读。

对于外部链接，应显示域名或来源名称，让用户知道即将跳转的位置。若链接来自第三方，建议使用“外部链接”样式或图标进行区分。

## 4. 搜索功能怎么设计

搜索应服务于产品资料查找，不宜只搜索产品名称。建议搜索范围覆盖以下字段：

- 产品名称。
- 品牌。
- 型号。
- 分类。
- 简短描述。
- 关键词标签。
- 文档标题或资料名称。

搜索框建议放在列表页顶部，并在移动端保持容易触达。搜索结果应即时反馈匹配数量，例如“找到 12 个产品”。当没有结果时，应展示空状态，提示用户尝试减少关键词、切换分类或清空筛选。

搜索交互建议：

- 输入关键词后实时过滤，或点击搜索按钮后过滤。
- 支持清空关键词。
- 搜索词与分类筛选同时生效。
- 保留当前筛选条件，避免搜索后分类状态丢失。
- 可在结果条目中突出匹配字段，但不作为第一阶段必须功能。

如果后续数据规模较小，前端可以读取静态 JSON 后在浏览器中完成搜索。若数据规模明显增长，再考虑服务端搜索或构建索引。

## 5. 分类筛选怎么设计

分类筛选应帮助用户缩小产品范围。建议优先支持单选分类，后续再扩展多选筛选。第一阶段不要设计过复杂的筛选系统，否则会增加数据维护成本。

建议分类筛选包含：

- 全部产品。
- 一级分类，例如产品大类或应用领域。
- 二级分类，如果数据结构已经稳定。
- 品牌筛选。
- 资料状态筛选，例如有 PDF、有图片、有说明文档。

桌面端可以在左侧或顶部使用筛选区域；移动端建议使用折叠面板或抽屉式筛选，避免占用过多首屏空间。

筛选结果应有明确反馈：

- 显示当前选中的分类或条件。
- 显示匹配产品数量。
- 提供“一键清除筛选”入口。
- 空结果时给出可操作提示。

分类名称应尽量来自静态 JSON 中的标准字段，避免前端硬编码一套分类文案。若存在别名或中英文名称，建议在数据层维护统一映射，前端只负责展示。

## 6. 图片链接失效时如何兜底

图片失效是产品资料展示中常见问题，前端需要有清晰兜底方案，避免页面出现破图或布局塌陷。

建议兜底策略：

- 固定图片容器比例，防止图片加载失败导致卡片高度变化。
- 主图加载失败时显示统一占位图。
- 占位图中展示产品名称首字、分类图标或“暂无图片”提示。
- 详情页中若补充图片失效，应隐藏该图片项或显示错误状态。
- 图片列表全部失效时，展示统一空状态，而不是空白区域。

数据层也可增加图片状态字段，例如 `imageStatus`、`hasImage` 或 `fallbackImage`。前端展示时优先读取有效图片链接，若没有主图则尝试备用图，最后才显示占位图。

占位图风格应保持克制，重点是维持布局稳定和信息可读，不应让占位图比真实产品图片更抢眼。

## 7. 静态 JSON 数据如何被前端读取

后续如果采用静态 JSON 作为产品资料来源，前端读取方式应尽量简单、可部署、可追踪。

推荐数据组织方式：

- 将产品基础数据集中在一个 JSON 文件中。
- 每个产品使用稳定唯一 ID。
- 图片、PDF、说明文档等资料链接作为产品对象中的数组字段。
- 分类、品牌、标签等字段保持结构化，不只写在描述文本里。
- 保留 `source`、`updatedAt`、`createdAt` 等元信息。

前端读取静态 JSON 时，可以在页面加载阶段获取数据，再在客户端完成列表渲染、搜索和筛选。若使用静态站点构建，也可以在构建阶段读取 JSON 并生成页面。具体技术路线可由后续前端实现阶段决定，本研究不写具体代码。

需要注意：

- JSON 字段命名应稳定，避免频繁改动导致页面展示逻辑失效。
- 图片和文档链接建议使用相对路径或可长期访问的绝对 URL。
- 外部资源应记录来源，便于失效后追查。
- 静态 JSON 不应混入展示样式字段，展示样式应由前端页面统一控制。
- 如果产品很多，可按分类拆分 JSON，但需要保留统一索引文件。

## 8. 后续适合用什么页面结构

后续页面结构建议从简单、可维护的资料站开始，而不是一开始就做复杂应用。适合的结构如下：

- 首页或产品资料首页：展示搜索框、主要分类、最新或重点产品。
- 产品列表页：展示所有产品，支持搜索、分类筛选、排序。
- 产品详情页：展示单个产品的完整信息、图片和文档链接。
- 分类页：按分类聚合产品，适合从导航直接进入某一类资料。
- 资料索引页：集中列出所有 PDF、说明文档和外部资料，方便只找文件的用户。
- 维护说明页：说明数据来源、字段含义、资料更新方式和失效链接处理原则。

推荐的页面关系：

```text
产品资料首页
  -> 产品列表页
      -> 产品详情页
  -> 分类页
      -> 分类下的产品列表
  -> 资料索引页
      -> PDF / 图片 / 说明文档 / 外部链接
```

第一阶段可优先完成“产品列表页 + 产品详情页 + 静态 JSON 数据读取 + 基础搜索筛选”。第二阶段再增加资料索引页、相关产品、资料完整度统计和失效链接维护提示。

## 总结建议

前端展示产品资料时，应把“用户快速找到产品”和“用户确认资料是否完整”作为核心目标。列表页负责浏览和筛选，详情页负责完整呈现，资料链接集中管理，图片失效要有稳定兜底。静态 JSON 数据适合当前阶段，但字段结构需要提前规范，避免后续页面实现时反复调整。

本研究建议后续采用轻量资料站结构，以产品列表页和产品详情页为核心，再逐步扩展分类页与资料索引页。
