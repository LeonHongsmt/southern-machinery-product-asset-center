# 产品资料数据结构研究

本文档用于规划产品资料的数据结构，服务于后续资料整理、前端展示、搜索筛选和来源追溯。当前只定义研究性 schema，不涉及业务代码、接口实现或爬虫实现。

## 设计目标

- 统一保存产品型号、产品名称、分类、资料文件、图片、说明文档和来源页面。
- 支持一个产品对应多个 PDF、多张图片、多个说明文档。
- 支持文件名中没有明确型号、来源资料不完整、后续人工补充的情况。
- 同一套字段可以落地为 JSON、CSV、Excel 三种格式。
- 前端读取时字段语义清晰，便于列表页、详情页、搜索和筛选使用。

## 字段总览

| 字段名 | 类型建议 | 是否必填 | 是否可为空 | 说明 |
| --- | --- | --- | --- | --- |
| product_model | 字符串 | 是 | 可为空 | 产品型号。没有明确型号时填空，并在 remarks 说明原因。 |
| product_name | 字符串 | 是 | 不可为空 | 产品名称，用于前端主标题和搜索。 |
| category | 字符串 | 是 | 不可为空 | 产品分类，例如加工中心、车床、磨床、配件等。 |
| file_name | 字符串 | 是 | 不可为空 | 主要资料文件名或原始文件名。若文件名没有型号，保留原文件名。 |
| file_type | 字符串 | 是 | 不可为空 | 主要文件类型，例如 pdf、image、manual、catalog、unknown。 |
| pdf_links | 数组 / 多值字段 | 否 | 可为空 | PDF 资料链接列表，可包含产品目录、规格书、宣传册等。 |
| image_links | 数组 / 多值字段 | 否 | 可为空 | 产品图片链接列表，可包含主图、细节图、应用图。 |
| manual_links | 数组 / 多值字段 | 否 | 可为空 | 说明书、操作手册、维护手册等链接列表。 |
| source_url | 字符串 | 是 | 不可为空 | 数据来源页面 URL，用于追溯和校验。 |
| description | 字符串 | 否 | 可为空 | 产品简介、用途、关键卖点或页面摘要。 |
| remarks | 字符串 | 否 | 可为空 | 人工备注，例如型号缺失、资料待确认、来源页面异常。 |
| created_at | 日期时间字符串 | 是 | 不可为空 | 记录创建时间，建议使用 ISO 8601 格式。 |
| updated_at | 日期时间字符串 | 是 | 不可为空 | 记录最后更新时间，建议使用 ISO 8601 格式。 |

## 必填字段

建议作为记录级必填字段：

| 字段名 | 必填原因 |
| --- | --- |
| product_model | 型号是产品识别核心字段；即使未知，也应保留字段并填空，避免不同格式缺列。 |
| product_name | 前端展示和人工识别必须依赖产品名称。 |
| category | 支持分类页、筛选器和导航结构。 |
| file_name | 保留原始资料文件名，方便溯源和文件管理。 |
| file_type | 前端需要根据类型决定展示方式和标签。 |
| source_url | 必须能追溯资料来源。 |
| created_at | 支持数据版本管理和导入记录审计。 |
| updated_at | 支持后续增量更新和前端缓存判断。 |

注意：`product_model` 是必填字段名，但字段值允许为空。这样可以兼容“资料存在但型号未知”的情况，同时保证 schema 稳定。

## 可以为空的字段

| 字段名 | 为空场景 |
| --- | --- |
| product_model | 文件名、页面标题、正文中都没有明确型号。 |
| pdf_links | 当前产品没有 PDF 资料，或 PDF 链接尚未整理。 |
| image_links | 当前产品没有图片，或图片链接尚未整理。 |
| manual_links | 当前产品没有说明书或手册。 |
| description | 页面没有可靠简介，或等待人工补充。 |
| remarks | 没有特殊备注时可以为空。 |

不建议为空的字段包括：`product_name`、`category`、`file_name`、`file_type`、`source_url`、`created_at`、`updated_at`。

## 多文件存储规则

一个产品可能对应多个 PDF、多张图片、多个说明文档。建议统一使用“列表”概念表达多值字段。

| 字段名 | 存储内容 | 排序建议 |
| --- | --- | --- |
| pdf_links | PDF 链接列表 | 优先放主产品目录，其次规格书、宣传册、其他 PDF。 |
| image_links | 图片链接列表 | 第一张作为默认主图，其余作为详情图或图库。 |
| manual_links | 说明文档链接列表 | 优先放操作手册，其次维护手册、安装手册、其他说明文件。 |

多值字段的推荐规则：

- JSON 中使用数组保存多个链接。
- CSV 中使用分隔符保存多个链接，建议使用英文分号加空格分隔。
- Excel 中可以采用单元格内多行或英文分号分隔；如果后续需要更规范的数据关系，也可以拆成独立附件表。

## 文件名没有型号时的存储方式

如果文件名没有产品型号，不应强行从文件名中猜测型号。建议按以下规则处理：

| 场景 | product_model | file_name | remarks |
| --- | --- | --- | --- |
| 文件名没有型号，但页面有明确型号 | 填页面中的型号 | 保留原始文件名 | 注明型号来自页面内容。 |
| 文件名和页面都没有型号，但有产品名称 | 留空 | 保留原始文件名 | 注明“原始文件名未包含型号，来源页面未发现明确型号”。 |
| 文件名疑似包含简称或系列名 | 填确认后的型号或留空 | 保留原始文件名 | 注明疑似信息，等待人工确认。 |
| 一个文件对应多个型号 | 可填主型号或系列名 | 保留原始文件名 | 在 remarks 中列出关联型号或说明“覆盖多个型号”。 |

核心原则：

- `file_name` 永远保存原始文件名，不为了补型号而改写。
- `product_model` 只保存可确认的型号。
- 无法确认的信息放入 `remarks`，不要混入正式字段。

## JSON 格式设计

JSON 适合前端直接读取，也适合保存数组型字段。推荐将每个产品保存为一个对象，多条产品记录组成数组。

| 字段名 | JSON 表达方式 |
| --- | --- |
| product_model | 字符串；未知时为空字符串或 null，建议统一为空字符串。 |
| product_name | 字符串。 |
| category | 字符串。 |
| file_name | 字符串。 |
| file_type | 字符串。 |
| pdf_links | 字符串数组；没有内容时为空数组。 |
| image_links | 字符串数组；没有内容时为空数组。 |
| manual_links | 字符串数组；没有内容时为空数组。 |
| source_url | 字符串。 |
| description | 字符串；没有内容时为空字符串。 |
| remarks | 字符串；没有内容时为空字符串。 |
| created_at | 日期时间字符串。 |
| updated_at | 日期时间字符串。 |

JSON 推荐约定：

- 多链接字段使用数组，不使用逗号拼接字符串。
- 空的多链接字段使用空数组，便于前端直接判断数量。
- 日期时间使用统一格式，例如 `YYYY-MM-DDTHH:mm:ssZ`。
- 如果未来需要更精细的附件信息，可以把链接数组升级为附件对象数组，但当前阶段保持简单字段即可。

## CSV 格式设计

CSV 适合批量导入、人工校对和简单数据交换。由于 CSV 单元格不天然支持数组，建议多链接字段使用固定分隔符。

| CSV 列名 | 填写规则 |
| --- | --- |
| product_model | 文本；未知时留空。 |
| product_name | 文本；不可留空。 |
| category | 文本；不可留空。 |
| file_name | 文本；保留原始文件名。 |
| file_type | 文本；统一小写，例如 pdf、image、manual。 |
| pdf_links | 多个链接用英文分号加空格分隔；没有则留空。 |
| image_links | 多个链接用英文分号加空格分隔；没有则留空。 |
| manual_links | 多个链接用英文分号加空格分隔；没有则留空。 |
| source_url | 文本；不可留空。 |
| description | 文本；可留空。 |
| remarks | 文本；可留空。 |
| created_at | 日期时间文本。 |
| updated_at | 日期时间文本。 |

CSV 推荐约定：

- 列顺序固定，便于后续导入。
- 链接分隔符统一为英文分号加空格，避免和 URL 中常见字符冲突。
- 文本中如果包含换行、英文逗号或引号，需要按 CSV 标准进行转义。
- CSV 中的空数组表现为空单元格，导入到 JSON 时再转换为空数组。

## Excel 格式设计

Excel 适合人工维护、筛选和备注。建议至少设计一个主表；如资料量增大，可增加附件明细表。

### 单表设计

| Excel 列名 | 填写规则 |
| --- | --- |
| 产品型号 | 对应 product_model，未知时留空。 |
| 产品名称 | 对应 product_name。 |
| 分类 | 对应 category。 |
| 文件名 | 对应 file_name。 |
| 文件类型 | 对应 file_type。 |
| PDF 链接 | 对应 pdf_links，多链接可用单元格内换行或英文分号分隔。 |
| 图片链接 | 对应 image_links，多链接可用单元格内换行或英文分号分隔。 |
| 说明文档链接 | 对应 manual_links，多链接可用单元格内换行或英文分号分隔。 |
| 来源 URL | 对应 source_url。 |
| 描述 | 对应 description。 |
| 备注 | 对应 remarks。 |
| 创建时间 | 对应 created_at。 |
| 更新时间 | 对应 updated_at。 |

### 双表设计

当附件很多时，可以拆成“产品主表”和“资料附件表”。

| 表名 | 作用 |
| --- | --- |
| 产品主表 | 保存 product_model、product_name、category、source_url、description、remarks、created_at、updated_at 等产品级信息。 |
| 资料附件表 | 保存 file_name、file_type、link、link_group、product_model、source_url 等附件级信息。 |

附件表中的 `link_group` 可用于区分 `pdf_links`、`image_links`、`manual_links`。这种方式更适合后续做关系型数据库或复杂筛选，但维护成本高于单表。

## 前端读取建议

前端读取 schema 时，应把每条记录视为一个产品资料对象。

| 前端位置 | 推荐读取字段 |
| --- | --- |
| 产品列表页 | product_name、product_model、category、image_links、description。 |
| 产品详情页 | 全部字段，重点展示 pdf_links、image_links、manual_links、source_url。 |
| 分类筛选 | category。 |
| 搜索 | product_model、product_name、file_name、description、remarks。 |
| 资料下载区 | pdf_links、manual_links、file_name、file_type。 |
| 来源说明 | source_url、created_at、updated_at、remarks。 |

前端处理规则建议：

- `image_links` 第一项作为默认主图；为空时展示占位图或隐藏图片区域。
- `pdf_links` 和 `manual_links` 按数组长度渲染多个下载入口。
- `product_model` 为空时，前端可以显示“型号待确认”或仅展示产品名称。
- `description` 为空时，不显示空白描述区。
- `remarks` 默认不作为营销内容展示，可放在后台、调试视图或资料来源说明中。
- `source_url` 可作为“资料来源”链接，帮助用户或运营人员追溯原始页面。
- `updated_at` 可用于判断资料是否较新，也可在详情页显示“资料更新时间”。

## 数据质量建议

| 检查项 | 建议 |
| --- | --- |
| 型号一致性 | 同一产品型号应尽量保持大小写和连字符格式一致。 |
| 分类命名 | 分类应使用固定词表，避免同义词造成筛选混乱。 |
| 链接有效性 | 后续导入前应检查链接是否可访问。 |
| 文件类型 | file_type 使用固定枚举值，避免 PDF、Pdf、pdf 混用。 |
| 来源追溯 | source_url 必须指向可验证的来源页面，而不是仅指向下载文件。 |
| 时间格式 | created_at 和 updated_at 使用统一时区和统一格式。 |

## 推荐最小记录

一条最小可用记录至少应具备：

| 字段名 | 最小要求 |
| --- | --- |
| product_model | 可为空，但字段必须存在。 |
| product_name | 必须有值。 |
| category | 必须有值。 |
| file_name | 必须有值。 |
| file_type | 必须有值。 |
| pdf_links | 可为空数组。 |
| image_links | 可为空数组。 |
| manual_links | 可为空数组。 |
| source_url | 必须有值。 |
| description | 可为空。 |
| remarks | 可为空。 |
| created_at | 必须有值。 |
| updated_at | 必须有值。 |

## 结论

推荐以 JSON 作为前端和系统读取的主格式，因为它天然支持多链接数组；CSV 和 Excel 作为人工整理、批量校对和数据交换格式。三种格式应保持字段语义一致，区别只在于多值字段的表达方式。

本 schema 的关键约定是：产品级信息放在固定字段中，多资料链接放在列表字段中，无法确认的信息进入 remarks，所有记录必须保留 source_url 以便追溯。
